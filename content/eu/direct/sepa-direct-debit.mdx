SEPA Direct Debit (SDD) is a pan-European payment scheme that enables organisations to collect euro-denominated payments from a payer’s account, provided the payer has authorised the collection in advance. The scheme is defined and maintained by the <ExternalLink href="https://www.europeanpaymentscouncil.eu/what-we-do/sepa-direct-debit">European Payments Council (EPC)</ExternalLink> and is available across SEPA-participating countries.

This page describes **SEPA Core Direct Debit only**, where ClearBank acts as the **debtor bank**. In this model, ClearBank receives direct debit instructions from the scheme and processes the resulting payment.

SEPA Direct Debits are initiated by the payee (creditor) rather than the payer. This makes the scheme suitable for recurring payments, subscriptions, and variable-amount collections where a prior authorisation exists.

### SEPA Direct Debit mandates

An SDD mandate is the payer’s authorisation for a creditor to initiate direct debit collections against the payer’s account. A mandate provides a legal basis for each collection submitted under the scheme.

From an integration perspective, mandates are **referenced rather than transmitted** as part of the payment flow: ClearBank receives mandate details from the scheme but does **not** create, store, validate, or manage mandates directly.

You are responsible for ensuring that:

- A valid mandate exists for each payment collection.
- Mandate details can be produced in the event of a dispute or scheme enquiry.
- Mandates are managed in accordance with the EPC's SEPA Core Direct Debit rulebook.

<Callout colour="blue">  
<strong>Note:</strong> ClearBank does not validate mandate existence or correctness when processing SDD collections. Responsibility for mandate management remains with the creditor.
</Callout>

### Pre-notification

Under the SEPA Core Direct Debit scheme, creditors are required to notify debtors in advance of an upcoming direct debit collection. This is known as **pre-notification**.

Pre-notification is a scheme obligation between the creditor and the debtor and typically includes the collection amount and date. The timing of the notification is defined by the EPC rulebook unless an alternative agreement is in place.

ClearBank does not validate or enforce pre-notification requirements, and pre-notification is not represented in the API or event model.

### Payment cycles

SEPA Core Direct Debit payments are processed using **batch-based clearing and settlement cycles**, rather than real-time processing. ClearBank receives debit collection instructions from the scheme and applies the resulting debit to the debtor account.

At a high level, a SEPA Core Direct Debit follows these stages:

1. **Collection received**  
   ClearBank receives an inbound direct debit instruction from the scheme.

2. **Payment processing**  
   The debit is applied to the debtor account in line with scheme processing rules.

3. **Settlement**  
   Funds are settled to the creditor via the scheme.

4. **Post-settlement outcomes**  
   Depending on scheme rules and reason codes, a collection may later be refused, returned, or refunded.

Because processing is not immediate, you should design your integration to be **event-driven** and rely on webhooks to track the outcome of each collection.

<Callout colour="blue">  
<strong>Note:</strong> Scheme timelines are defined by the EPC SEPA Core Direct Debit rulebook and may change. You should not rely on fixed day counts when modelling payment states.
</Callout>

## SEPA Direct Debit R-flows

### Overview

SEPA Direct Debit has three 'R-flows': Refuse, Return, and Refund. Each R-flow represents an outcome in the payment lifecycle. The safest way to reconcile R-flows is by using canonical payment identifiers.

### Return

A return is a request submitted by you after a SEPA Direct Debit has settled, instructing the bank to return the funds back to the debtor. This must be requested within the scheme’s permitted timeframe (up to 5 working days after settlement).

A return can be submitted when you determine that a settled SEPA Direct Debit should be returned to the debtor, for example due to insufficient funds or account restrictions. You must specify a valid return reason code when submitting the request.

Returns may be requested and received up to 5 working days after the date of settlement.

Returns have standard reason codes to help you resolve and communicate the underlying reason for a return:

#### Return reason code table

| Reason Code | Description                         |
| ----------- | ----------------------------------- |
| `AC06`      | BlockedAccount                      |
| `AM04`      | InsufficientFunds                   |
| `BE05`      | UnrecognisedInitiatingParty         |
| `MD01`      | NoMandate                           |
| `MD07`      | EndCustomerDeceased                 |
| `MS02`      | NotSpecifiedReasonCustomerGenerated |
| `MS03`      | NotSpecifiedReasonAgentGenerated    |
| `RR04`      | RegulatoryReason                    |

Fields must be populated with an appropriate return reason code and the original payment ID set out in [POST /payments/sepa-dd/v1/emulator/inbound-direct-debits](#simulate-an-inbound-sdd-payment-request).
